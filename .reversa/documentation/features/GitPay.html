Requisitos Gerais & Regras de Negócio

Módulo GitPay


Módulo de checkout e processamento de pagamentos Bitcoin descentralizado e serverless.


Visão Geral

O GitPay é um processador de pagamentos Bitcoin serverless e descentralizado executado inteiramente no navegador do cliente (browser-side). O sistema elimina intermediários custodiantes permitindo que merchants recebam fundos diretamente em seus endereços Bitcoin gerados de forma determinística a partir de suas próprias chaves BIP32 estendidas (xpub/ypub/zpub), publicando metadados cifrados das faturas na rede descentralizada Nostr.


Responsabilidades

  • Derivação de Endereços: Gerar endereços Bitcoin de forma determinística (BIP32/SLIP-0132) em caminhos de recebimento baseados em índices sequenciais.
  • Cifragem de Faturas: Garantir a privacidade financeira do merchant criptografando localmente os dados sensíveis das faturas.
  • Integração com Nostr: Utilizar a rede Nostr como um ledger de publicação/sincronização de faturas (Kind 30023) e recibos de pagamento (Kind 23001).
  • Verificação de Rede: Monitorar e confirmar pagamentos diretamente na blockchain e mempool do Bitcoin através da API do mempool.space.
  • Doação Probabilística: Direcionar uma taxa de comissão de suporte de 1% das faturas geradas de forma aleatória para a chave do desenvolvedor.

Regras de Negócio

  • Derivação Baseada em Prefixo 🟢: O tipo de endereço gerado é determinado automaticamente pelo prefixo da chave pública estendida (zpub/vpub -> Bech32 Native SegWit, ypub/upub -> P2SH-P2WPKH Nested SegWit, xpub/tpub -> P2PKH Legacy).
  • Cifragem Local 🟢: As faturas devem ser criptografadas localmente via AES-GCM (WebCrypto) utilizando uma chave simétrica de 256 bits derivada deterministicamente do hash SHA-256 da chave mestre do merchant e o índice da fatura.
  • Fallback de Cifragem 🟢: Em conexões inseguras HTTP ou locais onde a API WebCrypto Subtle está indisponível, o sistema utiliza um algoritmo de cifragem simétrica XOR (`xorEncryptDecrypt`) como barreira de fallback.
  • Tolerância de Pagamento (Threshold) 🟢: A fatura é considerada liquidada (`paid`) se o total recebido em blockchain (confirmado + mempool) for maior ou igual ao limite de tolerância configurado (padrão de 99.5% do valor solicitado).
  • Doação ao Desenvolvedor (Taxa de 1%) 🟢: O sistema substitui o XPUB de recebimento do merchant pelo do desenvolvedor em 1% das faturas geradas, sorteado via `Math.random()`.
  • Modo Offline e PWA 🟢: O sistema deve registrar um Service Worker para cache local IndexedDB e permitir o funcionamento offline básico.

Requisitos Funcionais


| ID | Requisito | Prioridade | Critério de Aceite |

|----|-----------|-----------|-------------------|

| RF-01 | Derivação BIP32 / SLIP-0132 | Must | Gerar endereços compatíveis (P2WPKH, P2SH-P2WPKH ou P2PKH) mapeados pelo prefixo da chave. |

| RF-02 | Cifragem AES-GCM local de metadados | Must | Criptografar descrição, valor e timestamps antes de enviar ao relay Nostr. |

| RF-03 | Fallback XOR para cifragem | Should | Mudar automaticamente para XOR se window.crypto.subtle for nulo. |

| RF-04 | Publicação de fatura no Nostr (Kind 30023) | Must | Publicar evento assinado com a tag d contendo o hash do endereço Bitcoin. |

| RF-05 | Escuta de recibos no Nostr (Kind 23001) | Must | Monitorar relays por eventos Kind 23001 contendo a tag do ID da fatura para iniciar checagem rápida. |

| RF-06 | Verificação via API mempool.space | Must | Obter soma das transações no endereço de rede e aplicar limite de tolerância (Threshold). |

| RF-07 | Direcionamento probabilístico de doação | Should | Em 1% das faturas (sorteio 0 a 100 < 1.0), aplicar a chave do desenvolvedor. |

Critérios de Aceitação



Dado que o merchant possui uma chave zpub configurada

Quando uma nova fatura com índice 10 é gerada

Então o sistema gera um endereço Bech32 P2WPKH derivado do caminho m/0/10



Dado que o valor solicitado é 100.000 satoshis com tolerância de 0.5% (Threshold = 99.500 sats)

Quando a blockchain acusa o recebimento de 99.600 satoshis para o endereço correspondente

Então a fatura é marcada como paga (paid) e o recibo é emitido



Dado que o valor solicitado é 100.000 satoshis com tolerância de 0.5% (Threshold = 99.500 sats)

Quando o recebimento acusa 95.000 satoshis

Então a fatura permanece pendente e exibe uma mensagem com o valor restante exigido de 5.000 satoshis

Módulo GitPay, Design Técnico


Módulo central de checkout e processamento de pagamentos Bitcoin descentralizado.


Interface


Como uma aplicação serverless client-side baseada em frontend web estático e bibliotecas criptográficas compiladas/embarcadas, o GitPay não expõe endpoints HTTP de servidor próprios. Em vez disso, expõe funções utilitárias em JavaScript nativo e interações com APIs externas.
