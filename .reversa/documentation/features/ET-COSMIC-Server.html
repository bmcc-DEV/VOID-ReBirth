ET-COSMIC-Server


Requisitos funcionais e de negócio do servidor central do ecossistema ET-COSMIC.


Visão Geral

O módulo ET-COSMIC-Server é a camada centralizadora e orquestradora do backend HTTP do ecossistema ET-COSMIC. Ele expõe uma API REST robusta que integra de forma híbrida os pagamentos via canais Lightning Network, serviços de criptografia pós-quântica (PQC-as-a-Service), emuladores de física exótica (LUSUS, AQRE, ETERNET, Isossupra, IMC), malha P2P/CDN (SilentMesh) e controle financeiro de tiers e reputação.


Responsabilidades

  • Inicializar a API HTTP baseada no Express e gerenciar políticas de segurança CORS. 🟢
  • Prover resiliência a falhas na conexão ao nó LND local com desvio para modo de simulação. 🟢
  • Servir assinaturas e chaves de criptografia pós-quântica ML-KEM/ML-DSA. 🟢
  • Roteamento unificado RPC via `/api/void` para serviços internos e de malha. 🟢
  • Gerenciar pools de recursos, conciliação de SLAs e renovação automática de tiers. 🟢

Regras de Negócio

  • Fallback de Pagamento: Se a URL do REST LND estiver indisponível e a variável `LND_FALLBACK_SIM` for verdadeira, ativa a simulação em memória das invoices. 🟢
  • PQC-as-a-Service Monetizado: Cada requisição feita nos endpoints de criptografia ML-KEM ou ML-DSA debita `1 Satoshi` da API Key do desenvolvedor. Se o saldo zerar, retorna o erro HTTP `402 Payment Required`. 🟢
  • SLA e Prova de Uptime: Provedores de recursos computacionais devem reportar provas criptográficas periódicas de uptime para receber os incentivos acordados em SLA. 🟢

Design Técnico & Arquitetura

ET-COSMIC-Server, Design Técnico


Especificação técnica de design do servidor central HTTP.


Interface


Endpoints HTTP expostos


| Método | Caminho | Entrada | Saída | Status codes |

|--------|---------|---------|-------|--------------|

| POST | /api/pqc/v1/kem | API Key e segredos | Par de chaves ou segredo Kyber | 200, 402, 500 |

| POST | /api/void | RPC payload | RPC response | 200, 400, 500 |

| GET | /api/lightning/status/:id | id: string | Status da invoice | 200, 404, 500 |


Fluxo Principal

1. O servidor Express inicializa configurando middlewares (CORS, body parser, log).

2. Tenta conexão à API LND REST. Em caso de falha, habilita o modo fallback simulated se configurado.

3. Carrega o banco JSON de API keys (db-apikeys.json).

4. Inicia listeners de rotas REST e RPC:

  • Rotas de PQC sob o middleware `requireApiKey`.
  • Roteador RPC central `/api/void`.
  • Rotas de controle de pools de recursos e SLA.

5. Inicializa os temporizadores Cron (renovação de tiers e auditorias de SLA).


Dependências

  • `pqcService`: resolvedores criptográficos e banco de chaves.
  • LND Node (ou simulador local).
  • Framework Express.

Estado Interno

  • `simulatedInvoices`: Mapa de invoices em memória se o LND estiver off-line.
  • `apiKeys`: Cache em memória das chaves dos desenvolvedores para evitar leitura excessiva em disco.
